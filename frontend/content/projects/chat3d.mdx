depth:
  problem: "No current LLM generates reliable Build123d code out of the box. Frontier models hallucinate class names, pass arguments in the wrong order, and produce code that looks reasonable but generates empty geometry. The gap is training data — Build123d is too niche for pre-training to cover."
  approach: "Built a 5-stage agent pipeline: spec generation, technique research via semantic search, agent-based code generation with tool-use loops, multi-agent decomposition for complex prompts, and 3-track evaluation (code assertions, code review LLM, VLM visual evaluation). Created a workbench with 1,100 curated prompts across 11 complexity categories to generate training data for fine-tuning."
  tradeoffs:
    - decision: "Agent scaffolding vs fine-tuning"
      alternatives:
        - "Complex agent pipeline with frontier models"
        - "Fine-tuned smaller model with simpler pipeline"
      reasoning: "Started with agent scaffolding to compensate for model knowledge gaps, but experiments showed a 120B open-weight model matches Opus at 1/5 the cost. Building toward fine-tuning to put domain knowledge into the weights rather than compensating with tooling."
    - decision: "Multi-track evaluation design"
      alternatives:
        - "Single LLM-based evaluation"
        - "Three independent tracks (code assertions, code review, VLM)"
      reasoning: "Each evaluation track has different failure modes. Code assertions catch dimensional errors but miss visual ones. VLM catches visual problems but can't verify precise measurements. Combining tracks with adaptive weighting produces more reliable quality scores."
    - decision: "Data flywheel architecture"
      alternatives:
        - "Manual training data curation"
        - "Automated generate-evaluate-approve pipeline"
      reasoning: "Automated pipeline with human review for edge cases scales to thousands of examples. User behavior (positive ratings, downloads) feeds back into training data through a curation pipeline."
  outcomes:
    - metric: "Model Benchmark"
      value: "120B matches Opus"

    - metric: "Training Dataset"
      value: "987 auto-approved"

    - metric: "Pipeline Success Rate"
      value: "90% with Opus"
      context: "90% success rate with Claude Opus, 75% with open-weight models at fraction of cost"
  reflections: "The key insight is that on niche domains, model size matters far less than domain knowledge. Agent scaffolding compensates for missing knowledge at the cost of tokens and latency — fine-tuning promises to close that gap permanently. The methodology generalizes to any domain where LLMs are almost good enough but fail on specifics."
